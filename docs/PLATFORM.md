# AgentOS Platform — сводный план

> Статус: draft, 2026-07-05. Итог обсуждения архитектуры платформы.
> Детальная спецификация протокола выносится в отдельную репу:
> https://github.com/try-agent-os/protocol (пишется параллельно).

## Видение

Платформа, где люди легко запускают своих Claude Code-агентов: готовые агенты
с мультиконнектором (MCP), каналы общения (Telegram, веб, мобильное приложение),
состояние в GitHub/S3, серверы поднимаются по кнопке. Текущий telegram-mcp —
зародыш приёмника (data plane) этой платформы.

## Архитектура: две части

### Часть 1 — единый бекенд (control plane)

Все клиенты — веб, Telegram-бот, Mini App, мобильное приложение — подключаются
к одному главному бекенду. Он же авторизационный сервер.

- **Стек**: NestJS turborepo (база — `novostudiotech/nestjs-fullstack-turborepo-starter`
  + auth из `nestjs-starter` + прод-наработки symoditi/platform MCP).
- **Модули**: `auth`, `telegram` (grammY — порт из этого репо), `gateway`
  (WS-сервер для приёмников), `mcp-router` (multi-router, подключаемые кастомные MCP),
  `provisioning`, `vault` (секреты).
- **Роутер**: таблица маршрутизации user→agent, очередь недоставленных сообщений.
  БД — Postgres (App Platform/managed; ephemeral FS исключает SQLite на роутере).

### Часть 2 — серверы пользователей (data plane)

Docker-образ с приёмником (receiver) на сервере каждого пользователя:

- подключается к роутеру **только outbound** по WebSocket (никаких входящих портов,
  NAT не мешает);
- транслирует команды в Claude Code: Agent SDK-сессии (`session-runner.ts`) +
  интерактивные tmux PTY;
- управляет MCP-подключениями, зависимостями, логин-флоу;
- один и тот же образ работает в двух режимах: `PAIR_TOKEN` задан → агент при
  центральном роутере; `TELEGRAM_BOT_TOKEN` задан → автономный self-host
  (сегодняшний продукт).

**Инварианты**:
1. Managed и self-hosted агент неразличимы для роутера.
2. Платформа не хранит credentials пользователя — только транзитит.
3. Конверт протокола не привязан к Telegram-типам (Telegram — один из каналов).

## Идентичность

Корень — внутренний `users.id`. Внешние привязки — таблица `identities`:

| Identity | Роль | Механизм |
|---|---|---|
| Google | обязательный email, вход в веб | OAuth |
| GitHub | вход + доступ к репо-состоянию агентов | OAuth |
| Telegram | бот, Mini App, вход в веб | Login Widget (HMAC) / initData / `from.id` |

Telegram **не отдаёт email** — связка аккаунтов живёт только в нашей БД:
one-time-токены и deep-link (`t.me/bot?start=TOKEN`) в обе стороны.
Текущий `access.ts` (allowlist/pending/deny) остаётся как модерация/анти-спам.

## Агент = папка = GitHub-репозиторий

Идентичность агента — его репо: файлы, инструкции, CLAUDE.md, скиллы, код.

- **Запуск**: контейнер поднимается → ФС гидрируется из GitHub-репы агента →
  восстановление рабочего состояния из S3 (Litestream-реплика SQLite).
- **Работа**: агент создаёт файлы, коммитит; синк обратно в GitHub + S3-бэкап.
- **Раскладка состояния**:
  - блочный **volume** — горячее состояние: `~/.claude` (credentials), SQLite,
    git checkout. S3 как рабочую ФС не монтировать (SQLite/git ломаются
    на S3-семантике);
  - **S3/Spaces** — блобы (медиа, артефакты), Litestream-реплика, бэкапы;
  - **GitHub** — переносимый «домашний каталог» агента, версии, аудит.
- Пользователь может «зайти в агента» и инспектировать файлы как ОС (Files UI).
- Каждый агент независим, со своим приёмником.

## Provisioning

`/connect` в боте (или кнопка в консоли) → платформа программно поднимает контейнер:

- **Ориентир — Fly.io Machines**: microVM из docker-образа за ~секунды, stop/start
  (спящий агент почти бесплатен), volumes, посекундный биллинг.
- Альтернативы: AWS ECS/Fargate (если жёстко в AWS ради S3), Hetzner/DO droplets
  (дешёвые VM, но флот-менеджмент), E2B/Modal/Daytona (vendor lock).
- Флоу: pairing token (TTL 15 мин) → приёмник дозвонился → обмен на долгоживущий
  agent credential → init (hydrate FS, deps, сервисы) → `ready`.

## Система зависимостей

Декларативный манифест (YAML, дух Brewfile/package.json) + lockfile: системные
программы агента — ffmpeg, whisper, обработка видео/аудио/текста, что угодно.

Три флоу:
1. установка недостающего при первом старте образа;
2. генерация prebaked-образов со всем предустановленным;
3. **агент сам дописывает зависимость в манифест → перезапуск → зависимость
   доступна** (self-service механизм).

Формат и lifecycle — в протокольной репе (раздел Dependency Manifest).

## Логин в Claude Code

Логин — свойство **сервера**, не сессии: SDK-сессии, интерактивный `claude` в tmux
и MCP читают один `$CLAUDE_CONFIG_DIR/.credentials.json`. Один логин — работают все.

Три пути (все — записи в одном vault-UI):

- **A (основной): проксированный OAuth.** Уже реализован и обкатан на хабе:
  `/login` → `login-flow.ts` → `scripts/claude-login-pipe.sh` (tmux-PTY: захват
  OAuth URL из pane, send-keys кода, поллинг ротации credentials + `auth status`,
  guard от гонок после рестарта). На платформе тот же механизм живёт в приёмнике
  и дёргается автоматически по auth-ошибке из SDK-потока; `auth_url`/`auth_code` —
  просто типы сообщений протокола.
- **B: готовый `claude setup-token`** — пользователь генерит локально, вставляет
  в secrets консоли → env контейнера.
- **C: `ANTHROPIC_API_KEY`** — API-биллинг, fallback без подписки.

UX-улучшения к текущему флоу: inline-кнопка вместо голого URL; код — reply на
конкретное сообщение (не «следующий текст в чате»); удалять сообщение с кодом
после сабмита; автотриггер вместо ручного `/login`.

Credentials остаются только на volume пользователя. ToS-позиция: подписка
пользователя работает на его выделенной машине, платформа — хостинг окружения.

## Secrets и Files (веб + Mini App)

Обе поверхности доступны и из веба, и из Mini App (текущий `src/console/` — зародыш UI):

- **Secrets**: статус Claude-авторизации (через приёмник), OAuth-токены коннекторов,
  API-ключи. Значения write-only (показывать маску), ротация, удаление.
- **Files**: браузер S3-бакета + git-workspace агента (последние коммиты, файлы).

## Маппинг текущего кода на платформу

| Сейчас (telegram-mcp) | В платформе |
|---|---|
| `bot.ts` (grammY) | Channel connector Telegram в бекенде |
| `access.ts` | Модерация; identity уходит в `auth`-модуль |
| `login-flow.ts` + `claude-login-pipe.sh` | Логин-флоу приёмника (per-user) |
| `session-runner.ts` / `sdk-runner.ts` | Ядро приёмника |
| `src/console/` (Mini App) | UI платформы (веб + miniapp) |
| `tools.ts` (MCP) | Серверная половина мультиконнектора |
| `db.ts` (SQLite+FTS) | Локальная БД агента + Litestream в S3 |

## Этапы

1. **Приёмник как автономный docker-образ** из текущего кода (self-host одной
   командой — уже продукт). Логин-флоу внутри.
2. **NestJS-скелет**: auth (Google/GitHub/Telegram) + gateway + pairing.
   Схема БД сразу с `users.id` + `identities`.
3. **Provisioning** на Fly Machines + volume/S3/Litestream.
4. **Консоль**: Secrets/Files, кнопка логина в Claude, управление агентами.
5. **MCP multi-router** и каталог коннекторов (OAuth-флоу коннекторов через веб,
   токены — в vault, инжект в контейнер).

Протокол (конверт, pairing, deps-манифест, auth-флоу, JSON Schemas) —
https://github.com/try-agent-os/protocol.
